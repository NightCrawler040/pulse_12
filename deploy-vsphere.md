# 🐧 Полное руководство по развертыванию Pulse 12 в виртуальной среде (vSphere / Proxmox / Cloud VM)

Данная инструкция является официальным корпоративным стандартом для системных администраторов и DevOps-инженеров по подготовке виртуальной машины, развертыванию, настройке сетевой безопасности, стратегии Git-ветвления и резервному копированию платформы **Pulse 12 (с PostgreSQL 16)**.

---

## 🖥️ 1. Требования к Виртуальной Машине (VM Specifications)

Для стабильной работы системы с 12–15 активными сотрудниками (онлайн WebSocket-вещание при 60 FPS, кэширование запросов и ACID-транзакции PostgreSQL) рекомендуется выделить следующие ресурсы в гипервизоре (VMware vSphere / ESXi, Proxmox или Hyper-V):

| Параметр | Минимальные требования | Рекомендуемые (Production) | Назначение и пояснение |
| :--- | :--- | :--- | :--- |
| **Операционная система** | Ubuntu Server 22.04 LTS (64-bit) | **Ubuntu Server 24.04 LTS / Debian 12** | Стандарт корпоративного Linux-хостинга с долгосрочной поддержкой (LTS). |
| **Процессор (vCPU)** | 2 ядра | **2–4 ядра** | Разделение нагрузки: 1 ядро под Node.js/Express/Socket.io, 1 ядро под PostgreSQL 16 и фоновые процессы ОС. |
| **Оперативная память (RAM)** | 2 ГБ | **4 ГБ** | 4 ГБ гарантируют отсутствие своппинга (swap) при кэшировании БД в памяти и одновременной загрузке вложений сотрудниками. |
| **Диск (SSD / NVMe Storage)** | 20 ГБ | **30 ГБ** | ~8 ГБ под ОС Ubuntu и образы Docker, остальные ~20 ГБ под рост базы данных (`pulse12_postgres_data`) и хранение документов/файлов в `server/data/uploads/`. |
| **Сетевой адаптер (NIC)** | 1 Гбит/с (DHCP/Static) | **1 Гбит/с (Static IP)** | Режим **Bridged (Сетевой мост)** или привязка к корпоративному VLAN. ВМ должна иметь фиксированный IP в локальной сети LAN (например, `192.168.10.50`). |

### ⚠️ Важные настройки в гипервизоре (VMware vSphere):
1. **Синхронизация времени**: Обязательно установите пакет `open-vm-tools` внутри Ubuntu, чтобы время сервера всегда синхронизировалось с ESXi-хостом. Точное время критически важно для корректного таймстемпа аудит-логов (`🤖 Системный аудит`) и валидации JWT-сессий.
2. **Автовключение ВМ**: Настройте в vSphere параметр *VM Startup/Shutdown*, чтобы виртуальная машина автоматически включалась при перезапуске физического сервера.

## 🌿 2. Стратегия ветвления в Git (`test` ➔ `main`)

В репозитории проекта ([NightCrawler040/pulse_12](https://github.com/NightCrawler040/pulse_12)) настроена двухконтурная модель разработки:

```
[ Локальная разработка ] ──(push)──> [ Ветка 'test' ] ──(тестирование)──> [ Ветка 'main' (Production) ]
```

1. **Ветка `test` (Test / Staging Environment)**:
   - Предназначена для проверки новых функций, экспериментов с интерфейсом и тестирования миграций БД.
   - На тестовой виртуальной машине разворачивается именно ветка `test`.
2. **Ветка `main` (Production Environment)**:
   - Стабильная боевая ветка. Сюда попадает только проверенный, скомпилированный код без ошибок TypeScript.
   - На основном корпоративном сервере vSphere работает ветка `main`.

## 🚀 3. Пошаговая инструкция по установке на новой ВМ Ubuntu

Подключитесь к вашей виртуальной машине по SSH (например, `ssh administrator@192.168.10.50`) и последовательно выполните следующие команды терминала:

### Шаг 1. Обновление ОС и установка базовых утилит
```bash
sudo apt update && sudo apt upgrade -y
sudo apt install -y curl git unzip htop ufw open-vm-tools ca-certificates gnupg
```

### Шаг 2. Установка Docker и Docker Compose V2
Используем официальный автоматизированный скрипт установки Docker:
```bash
# 1. Скачиваем и запускаем установщик Docker:
curl -fsSL https://get.docker.com -o get-docker.sh
sudo sh get-docker.sh

# 2. Устанавливаем плагин Docker Compose V2:
sudo apt-get install -y docker-compose-plugin

# 3. Добавляем текущего пользователя в группу docker (чтобы вводить команды без sudo):
sudo usermod -aG docker $USER
newgrp docker

# 4. Проверяем корректность установки:
docker --version
docker compose version
```

### Шаг 3. Клонирование репозитория на сервер
Создаем корпоративную директорию в `/opt` и клонируем проект:
```bash
sudo mkdir -p /opt/pulse12
sudo chown -R $USER:$USER /opt/pulse12
git clone https://github.com/NightCrawler040/pulse_12.git /opt/pulse12
cd /opt/pulse12
```

### Шаг 4. Выбор рабочей ветки
```bash
# Если вы настраиваете БОЕВОЙ сервер (Production):
git checkout main

# Если вы настраиваете ТЕСТОВЫЙ сервер (Staging / Test):
git checkout test
git pull origin test
```

### Шаг 5. Запуск контейнеров (PostgreSQL 16 + Pulse 12)
Запускаем автоматическую сборку и старт сервисов в фоновом режиме:
```bash
docker compose up --build -d
```
> **Что происходит в этот момент?**  
> 1. Docker поднимает изолированный контейнер `pulse12-postgres` с базой данных PostgreSQL 16.  
> 2. Компилируется и запускается Node.js сервер `pulse12-corporate`.  
> 3. При первом запуске система автоматически проверяет БД, создает все необходимые таблицы и загружает начальный корпоративный справочник (12 сотрудников, отделы, демо-задачи с аудит-логами).

### Шаг 6. Проверка работоспособности
Убедитесь, что оба контейнера находятся в статусе `Up` и `Healthy`:
```bash
docker compose ps
```
Для просмотра логов в реальном времени (выход по `Ctrl+C`):
```bash
docker compose logs -f pulse12-corporate
```
🎉 **Система запущена!** Откройте браузер на любом компьютере в офисе и введите адрес сервера: **`http://192.168.10.50/`** (замените на IP вашей ВМ).

## 🔒 4. Настройка сетевой безопасности и Брандмауэра (UFW)

Чтобы защитить сервер от посторонних подключений, оставляем открытыми только необходимые порты:
```bash
# 1. Открываем порт 22 (SSH для удаленного управления админом):
sudo ufw allow 22/tcp

# 2. Открываем порт 80 (HTTP веб-портал для всех сотрудников LAN):
sudo ufw allow 80/tcp

# 3. (Опционально) Открываем порт 3001, если используется прямое подключение по WebSocket:
sudo ufw allow 3001/tcp

# 4. Включаем брандмауэр:
sudo ufw --force enable
sudo ufw status verbose
```

## ♻️ 5. Процедура обновления (Как выкатывать новые релизы)

Когда разработчики внесли улучшения и запушили код в ветку `test` или `main`, обновление на сервере занимает всего 30 секунд без потери данных:

```bash
cd /opt/pulse12

# 1. Скачиваем свежий код из репозитория:
git pull origin main    # или git pull origin test (для тестовой ВМ)

# 2. Пересобираем контейнер приложения и перезапускаем его (база данных при этом не прерывает работу!):
docker compose up --build -d --no-deps pulse12-corporate

# 3. Очищаем старые неиспользуемые Docker-образы для освобождения места на диске:
docker image prune -f
```

## 🐘 6. Администрирование базы данных PostgreSQL 16 и Резервное копирование

Вся база данных надежно сохранена в защищенном Docker-томе `pulse12_postgres_data`.

### 💾 Создание резервной копии (Backup) вручную
Для создания полного дампа базы данных в SQL-файл:
```bash
mkdir -p /opt/pulse12/backups
docker exec -t pulse12-postgres pg_dump -U pulse12_admin pulse12 > /opt/pulse12/backups/pulse12_backup_$(date +%F_%H-%M).sql
```

### ⏰ Автоматический ночной бэкап (через Cron)
Чтобы система автоматически сохраняла резервную копию каждую ночь в **03:00**:
1. Откройте планировщик задач:
   ```bash
   crontab -e
   ```
2. Добавьте в конец файла строку (хранить бэкапы за последние 14 дней):
   ```cron
   0 3 * * * docker exec -t pulse12-postgres pg_dump -U pulse12_admin pulse12 > /opt/pulse12/backups/pulse12_$(date +\%F).sql && find /opt/pulse12/backups/ -name "*.sql" -mtime +14 -delete
   ```

### ♻️ Восстановление базы данных из бэкапа (Restore)
Если потребовалось откатить или восстановить базу из ранее сохраненного файла:
```bash
cat /opt/pulse12/backups/pulse12_backup_2026-07-05.sql | docker exec -i pulse12-postgres psql -U pulse12_admin -d pulse12
```

### 🔍 Вход в SQL-консоль psql
Для выполнения административных запросов напрямую внутри PostgreSQL:
```bash
docker exec -it pulse12-postgres psql -U pulse12_admin -d pulse12

# Полезные SQL-команды внутри консоли:
# \dt                             -- показать все таблицы в БД
# SELECT * FROM pulse_store;      -- посмотреть сохраненные данные
# SELECT id, name, role FROM pulse_store, jsonb_to_recordset(data->'users') AS x(id text, name text, role text); -- выборка пользователей
# \q                              -- выход из консоли psql
```

## 📊 7. Сводная таблица портов и путей

| Параметр | Значение | Описание |
| :--- | :--- | :--- |
| **Порт веб-портала** | `80` (HTTP) | Главный интерфейс для браузеров сотрудников |
| **Порт сервера API / WS** | `3001` (TCP) | Внутренний и внешний REST API и WebSocket Socket.io |
| **Порт PostgreSQL** | `5432` (TCP) | Реляционная СУБД (проброшена на хост при необходимости) |
| **Путь к проекту на ВМ** | `/opt/pulse12` | Корневая папка проекта с `docker-compose.yml` и кодом |
| **Папка загрузок файлов** | `/opt/pulse12/server/data/uploads` | Физическая директория на диске ВМ с документами и аватарками |
| **Том базы данных** | `pulse12_postgres_data` | Изолированный Docker Volume с файлами Postgres 16 |
